# Frida-based VM instrumentation for wrapper.node

## Purpose

Reverse-engineer the obfuscated VM inside `wrapper.node` so the sign function can be re-implemented in pure Python.

## Discoveries (2026-04-23 session)

### 1. Top-called functions during sign (Stalker call summary)

| Offset | Count | Nature |
|--------|------:|--------|

### 2. The 20-byte XOR stream lives in a 257-byte malloc'd buffer

Pattern `550504a20fd4f219c36087685573c224881743b7` (the XOR constants that are
applied to MD5(src) bytes) is written at offset 0 of a 257-byte buffer
returned by `malloc(257)`. The caller of malloc is reached through
`w+0x5cd56d0` and `w+0x5ccbaaf` (both are VM-obfuscated dispatch code).

### 3. The 20 bytes appear early in sign execution

Scanning heap at every 10 calls of `w+0x5cccffa` shows:
- Not present at call #100
- Present at call #110–120 (first detection)
- Freed later during sign

The short lifetime means simple post-hoc scans miss the buffer.

### 4. Buffer context around the 20 XOR bytes

```
offset -32..-1 (header): 00 00 00 00 01 00 00 00  0b 00 00 00 00 00 00 00
                         00 00 00 00 00 00 00 00  11 01 00 00 00 00 00 00
offset   0..19 (XOR):    55 05 04 a2 0f d4 f2 19  c3 60 87 68 55 73 c2 24
                         88 17 43 b7
offset  20..260 :        (mostly zeros — the 257-byte buffer is over-sized)
```

The `11 01 00 00 ...` preceding the XOR block decodes to u64 `0x111 = 273`,
likely a field tag or length marker, not an MD5/SHA1 size.

### 5. Existing trace generator template (`frida_complete_capture.py`)

The `/tmp/complete_trace_*.json` traces were generated by hooking an older
build of wrapper.node at these addresses:

- `base + 0x5F4DB51` — sign function entry
- `base + 0x5F56328` — VM dispatch instruction (inside sign)

At the dispatch instruction:
- `rax` holds the current opcode byte
- `rbp - 0x280` holds a pointer to the VM program counter (PC)
- `r14 + 0x10` holds a pointer to the 150-entry register array
  (each entry 8 bytes, but only the low 32 bits are used)

Current `wrapper.node` has sign at `base + 0x56D81D1` — offsets have shifted
between QQ versions. Finding the CURRENT dispatch instruction address is
the first step for live re-instrumentation.

## Open problem: who writes the 20 bytes?

Known: the 20-byte XOR stream is written somewhere between `malloc(257)` and
the first scan we can land. All attempts to catch the actual writer hit one
of these dead-ends:

1. `memcpy`/`memmove` don't carry the bytes (not a stdlib copy).
2. MemoryAccessMonitor fires only once per page (misses the subsequent stores).
3. Stalker transform with mem-write predicate got 0 events — either our
   predicate is too narrow or the writes are through non-obvious paths
   (SIMD stores, string functions without `mem` operand detection, etc.).
4. Heap addresses change each sign call, so hooks set up using one run's
   addresses don't fire in the next run.

## Recommended next tactics

1. **Use `rr` (record-and-replay)**: record one sign call and set hardware
   watchpoints at the exact XOR buffer address. `rr` is deterministic, so
   the address IS stable in replay.
2. **Patch wrapper.node** to redirect its own free() calls into a logger so
   the 20-byte buffer persists for post-hoc inspection.
3. **Statically reverse the VM dispatch tree** — the binary-tree `cmp $imm, %eax`
   dispatch is painful but fully static. Once decoded, every PC-hash maps to
   a handler offset, and handler semantics can be read off.
4. **Find CURRENT VM dispatch** in `wrapper.node`: the old script used
   `base + 0x5F56328`. Port this by pattern-matching the VM epilogue near
   the sign function's own dispatch loop at `base + 0x56D81D1`.
5. **Replay existing traces and watch each VM register as a pointer** — the
   150-entry register array at `r14+0x10` contains pointers. Dereferencing
   each at each step (via the trace hooks) would reveal buffer contents
   without needing post-hoc heap scanning.
